## Finding 1 — RISK-004/009/018/033/037/038: "Mitigated" claimed with no in-row implementation evidence (nit)

Severity: **nit** (honestly disclosed, not a silent upgrade).

Old text (representative, RISK-004): `**Mitigate** → PF AUTH — password storage with modern KDF …`
— i.e. the *decision to mitigate* + requirement mapping, with **no** per-row "status: implemented"
marker anywhere in the cell.

New text (RISK-004 Status): `Mitigated`; mitigation cell appends
`(Row carries only the original requirement mapping; no per-row implementation marker — see compaction log.)`

Why it is a nit and not a fact-changed finding: the new Status vocabulary defines *Mitigated* as
"a control now protects against the risk", which is a stronger claim than the old cell evidences for
these six rows. However, the compactor did not resolve this silently — it (a) appended the caveat to
every one of the six cells, (b) recorded each in the log's FLAGGED section, and (c) explicitly asks the
owner to confirm implementation status (esp. 004 = OPDS KDF/rate-limit/audit, 009 = FlareSolverr,
037 = CBL import). **RISK-037 is the sharpest**: RISK-024 states CBL reading-list import is *backlog B,
unbuilt*, so RISK-037's hardened-parser "Mitigated" is a design commitment for an unbuilt feature — its
cell says exactly that (`Design commitment only — CBL import is backlog and unbuilt per RISK-024`).
Arguably RISK-037 should be **Open** by parity with RISK-008, but the in-cell disclosure keeps it honest.
No action required of the compaction; the residual ambiguity is surfaced, not buried.
