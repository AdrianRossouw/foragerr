# site-regulated-story — design

## Context

A Claude Design exploration ("Foragerr Site", design project
`7c792029-57c1-46cc-b7f1-7be8ba30cb1b`, `Foragerr Site.dc.html`) defines the visual
direction: five pages in the app's own design system (dark surfaces, ant-green
accent, the frontend's tokens), with three hero variants and placeholder content.
The owner approved the shape and the regulated-story-first framing, chose hero
"A · Claim", and set the governing rule: every fact is generated from repo
artifacts or it does not appear. The design's placeholder data (invented releases,
pentest, SBOM, MIT, "1,040 tests") is explicitly layout-only.

The repo already produces the needed sources at every merge gate:
`docs/traceability/requirements-registry.md`, `docs/traceability/matrix.md`
(generated by `tools/trace.py`), `CHANGELOG.md`, `docs/security/risk-register.md`,
`docs/security/threat-model.md`, `docs/security/soup-register.md`, git tags, and
GitHub Releases. There are currently no GitHub Actions workflows; this change adds
the first.

## Goals / Non-Goals

**Goals:**
- Static site at `adrianrossouw.github.io/foragerr`, rebuilt on every push to main.
- Generator that parses the artifacts above and fails hard on any inconsistency.
- Faithful port of the design's layout/typography with truthful content.

**Non-Goals:**
- No JS framework, bundler, or CSS toolchain; no analytics; no custom domain yet.
- No CI test gate in this change (site build/deploy only).
- No changes to product code, `/release`, or the merge-gate checklist.

## Decisions

1. **Generator: single stdlib-only Python script (`site/build.py`).**
   The sources are pipe tables and structured markdown; parsing them needs regex,
   not a markdown library. This keeps the SOUP register free of new Python
   dependencies and matches the house style of `tools/trace.py` — whose
   registry/matrix parsing conventions are reused (same table shapes), though the
   site keeps its own parser to avoid coupling the merge gate to the site.
   *Alternative rejected:* Jekyll/MkDocs/Astro — each drags a toolchain and SOUP
   entries into a repo that otherwise builds with Python + npm only, for a site of
   five hand-shaped pages.

2. **Templates: HTML files with `${placeholder}` slots filled via
   `string.Template`; prose lives in the templates.** The story copy (method
   pillars, lifecycle, callout) is stable hand-written content — it is not a
   "fact" under FRG-SITE-001; all numbers/paths/history are injected. A build-time
   scan asserts templates contain no banned placeholder-era claims (FRG-SITE-006
   banned-phrase list lives in `site/banned_phrases.txt`).

3. **Design transcription, not design-system sync.** `site/static/site.css` ports
   the token values used by the dc.html (they are the frontend's tokens) as plain
   CSS custom properties. Fonts: system stack plus the repo's already-vendored
   Roboto Mono copied at build time — zero external requests, which also keeps the
   site CSP-clean.

4. **Hero trace card: exemplar requirement configured in `site/site.toml`**
   (parsed with stdlib `tomllib`), validated at build time against the registry and
   matrix; the card renders that row's spec path, test file, and newest commit.
   Owner picks the exemplar (default proposal: FRG-SER-019, the
   never-suppress-single-issue-wanted invariant — a real, story-friendly one).
   *Alternative rejected:* auto-picking a row — an unlucky pick (withdrawn/deferred
   row) would front the site with a weak example.

5. **Timeline: one entry per CHANGELOG release, newest first, grouped under minor
   version headers** (49 releases and growing; grouping keeps the page scannable
   without JS). Each entry shows the CHANGELOG's date, summary bullets, and the
   FRG IDs it cites, cross-checked against `git tag`. Per-release "test delta"
   figures from the design are dropped — the repo does not record them, and
   inventing them violates FRG-SITE-001.

6. **Stat strip = requirements count, traced-coverage (N of M requirements with
   tagged tests, from the matrix), releases count, milestones-shipped count.**
   Raw test totals are excluded by FRG-SITE-004.

7. **Deployment: `.github/workflows/pages.yml`**, triggered on `push:
   branches: [main]` (plus `workflow_dispatch`), single job: checkout →
   `python site/build.py --out _site` → `configure-pages` →
   `upload-pages-artifact` → `deploy-pages`. All actions pinned to full commit
   SHAs; permissions exactly `contents: read`, `pages: write`, `id-token: write`.
   Deploy-on-push (not on-release) means docs-only merges also refresh the site
   and `/release` stays untouched.
   *Alternative rejected:* committing built HTML to `/docs` on main — avoids a
   workflow but puts generated output under version control, which the project
   pattern (matrix.md excepted) avoids and which pollutes diffs at every gate.

8. **Screenshots on the Product page reuse the README screenshot set**
   (`tools/refresh-readme-shots.sh` output) copied at build time — no separate
   screenshot pipeline.

## Risks / Trade-offs

- [Matrix commit-window drift: `matrix.md`'s commit column is a rolling window, so
  hero-card commit hashes may age] → card renders the row's newest commit and links
  to the matrix itself as the authority; regenerated on every merge anyway.
- [First CI workflow = new supply-chain surface] → actions pinned by SHA and
  recorded in the SOUP register; least-privilege token; risk-register entry added
  in this change (docs/security update per FRG-PROC-006).
- [Public site could drift from positioning rules through future copy edits] →
  banned-phrase scan runs inside the build (fails deploy), and copy changes remain
  spec-governed under FRG-SITE-006.
- [Pages setting cannot be set from the repo] → one-time manual step by owner
  (Settings → Pages → Source: GitHub Actions); until then the workflow's deploy
  step fails visibly, deploying nothing stale.
- [stdlib-only parsing is stricter than a markdown library] → intentional: parse
  failures must fail the build, not degrade output (FRG-SITE-001).

## Migration Plan

New surface only; no data migration. Rollback = delete the workflow (site stops
updating) or disable Pages. Tags/releases untouched.
